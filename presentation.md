# GodMode: Hackathon Project Evaluation

**Project:** GodMode - Interactive Generative Storyteller

Here is a breakdown of our project based on the hackathon judging criteria.

### 1. Usage of Google Products (Vertex AI & Gemini)
Since we are new to these tools, our goal was to experiment with combining text, image, and audio APIs to see if we could build a complete interactive loop. 

* **Text Generation (Gemini Flash):** We used `gemini-3-flash-preview` / `gemini-2.0-flash` as the core story engine. We chose this because we needed a model that could stream text (so users don't get bored waiting) and reliably output structured JSON. The JSON lets us separate the narrative text from the A/B/C/D choices.
* **Image Generation (Imagen 4.0):** We used `imagen-4.0-fast-generate-001` via Vertex AI to create pictures for each chapter and covers for the story catalog. We wanted the app to feel like a visual novel or comic book. We take a "reveal" sentence generated by the text model and pass it to Imagen to get the picture.
* **Voice Narration (Gemini TTS with Streaming & Prefetching):** We used `gemini-2.5-flash-preview-tts` to read the story out loud. We picked it because it provides built-in voices (like Kore or Charon) that fit a dramatic storytelling style. To improve latency and make the experience feel seamless, we implemented background prefetching for the TTS audio of potential choices and chunked the PCM audio data via Server-Sent Events (SSE) so playback starts almost instantly.
* **Speech-to-Text (Gemini Multimodal):** We wanted users to be able to speak their choices. We record a `.webm` file from the browser microphone and send it to the Gemini model to transcribe what the user said, then map that to one of the choices.

### 2. "Fun Vibe" & User Engagement
We wanted to take the standard "choose your own adventure" format and make it feel more like a video game or a tabletop RPG session. 
* Instead of just reading, the user gets a custom image and voice narration for every turn. The new audio streaming ensures the narration feels responsive and snappy.
* By adding microphone support, users can literally speak their choice (e.g., saying "I choose option A"). 
* If users don't like the provided A/B/C/D choices, they can type their own custom action, meaning the story can go anywhere, which keeps it unpredictable and fun.

### 3. Technical Execution & Performance (Latency Optimization)
Building AI agents requires managing the inherent delays of calling multiple large models. We focused heavily on optimizing the user's perception of speed.
* **Predictive Prefetching:** Instead of waiting for the user to make a choice, the backend proactively generates the text and audio for all available options in the background (`prefetch.py`). When the user clicks an option, the app serves the cached result instantly.
* **Streaming Modalities:** By chunking both the generated text and the raw audio PCM bytes through Server-Sent Events (SSE), the user begins consuming the content milliseconds after it starts generating, rather than waiting for the entire block to finish.

### 4. Product Clarity & Loop Design
The concept is straightforward: it is an AI Dungeon Master.
* **The Flow:** The user selects a story from the catalog (like Hamlet) or types a custom prompt -> reads/listens to the generated chapter -> looks at the generated image -> makes a choice -> repeat.
* **State Management:** The backend maintains the story state, appending the user's choices and the AI's generated chapters sequentially, ensuring the ongoing context is maintained and narrative rules (e.g. keeping to a 10-chapter limit) are respected.

### 5. Social "Purpose" & Accessibility
* **Accessibility:** Because we implemented Text-to-Speech (the app reads to you) and Speech-to-Text (you can talk to the app), it can be played almost entirely hands-free. This makes it a good fit for users with visual or motor impairments.
* **Education:** It can take classic, older literature that students might find boring and turn it into an interactive game. Letting students explore "what if" scenarios in a book like *Hamlet* makes them actively participate in the reading.

---

### Potential 24-Hour Improvements (Future Work)
If we had another block of hackathon time, here is what we could feasibly improve:
1.  **Migrate Memory to an ADK:** Currently, we store the whole story history in a Python dictionary. We could implement an Agent framework to automatically summarize older chapters when context windows get too long.
2.  **Imagen Tool Calling:** Instead of hardcoding the image generation to happen after the text, we could give the Gemini agent the *ability* (Tool Calling) to trigger an image generation whenever a dramatic moment occurs in the story.
3.  **Database Migration:** Move active session storage from an in-memory dictionary to a simple SQLite or Redis instance so sessions survive server restarts.
